# 北京美食分析与推荐系统项目报告

## 一、项目概述

### 1.1 项目背景
随着互联网餐饮行业的快速发展，用户对美食推荐和数据分析的需求日益增长。本系统基于Django框架开发，旨在为北京地区的美食爱好者提供全面的店铺信息查询、数据分析、智能推荐和可视化展示功能。

### 1.2 项目目标
- 构建一个功能完善的美食信息管理系统
- 提供多维度数据分析和可视化展示
- 实现基于用户行为的智能推荐算法
- 集成地图服务，提供地理位置可视化
- 支持评论情感分析，挖掘用户真实反馈

### 1.3 技术栈
- **后端框架**: Django 5.2.10
- **数据库**: MySQL 8.0+
- **前端技术**: HTML5, CSS3, JavaScript, ECharts 5.4.3
- **地图服务**: 百度地图API v3.0
- **Python版本**: Python 3.10+
- **其他**: SimpleUI (Django Admin美化), PyMySQL

### 3.3 数据分析模块

#### 3.3.1 Top10排行榜 (`top10_rankings`)
**多维度榜单**:
1. **综合人气榜**: 归一化评论数和评分，权重0.4:0.6
2. **评分最高榜**: 按评分降序，评论数作为次要排序
3. **评论最多榜**: 按评论数降序
4. **人均最高榜**: 高端消费店铺
5. **人均最低榜**: 性价比之选
6. **米其林推荐榜**: 米其林认证店铺

**算法实现**:
```python
# 归一化处理
normalized_comments = comment_count / max_comments
normalized_score = shop_score / max_score
popularity_score = normalized_comments * 0.4 + normalized_score * 0.6
```

### 3.6 推荐系统模块

#### 3.6.1 推荐算法设计

**算法1: ItemCF协同过滤**
```python
# 步骤：
1. 找到收藏了用户已收藏店铺的其他用户
2. 统计这些用户还收藏了什么店铺
3. 按共同收藏用户数排序推荐
```

**算法2: 内容相似度**
```python
# 相似度计算：
- 同菜系类型
- 同区县位置
- 相似价格区间（±30%）
```

**算法3: 价格区间匹配**
```python
# 计算用户收藏店铺的平均价格
avg_price = sum(collected_prices) / len(collected_shops)
# 推荐价格在 [avg_price*0.7, avg_price*1.3] 范围内的店铺
```

**算法4: 热门推荐（冷启动）**
```python
# 综合排序：
popularity = shop_score * comment_count
```

#### 3.6.2 推荐理由生成
每个推荐都附带可解释的理由：
- "有X位用户同时收藏了您喜欢的店铺"（ItemCF）
- "与您收藏的X菜系店铺相似（X区县）"（内容相似）
- "价格相近（人均¥X，您常选¥Y）"（价格匹配）
- "人气爆款（评分X，Y条评论）"（热门推荐）

### 3.7 情感分析模块（管理员）

#### 3.7.1 任务管理
**任务类型**:
- 全部分析: 分析所有评论
- 增量分析: 只分析未分析的评论
- 重新分析: 重新分析已有评论
- 测试分析: 小批量测试

**任务状态**:
- pending: 等待中
- processing: 处理中
- completed: 已完成
- failed: 失败
- cancelled: 已取消

#### 3.7.2 分析流程
```
1. 管理员创建任务 → SentimentAnalysisTask
2. 任务加入Redis队列
3. Celery Worker处理任务
4. 批量分析评论（使用sentiment_core.py）
5. 更新评论的sentiment和sentiment_score字段
6. 更新任务进度和状态
7. 生成结果摘要
```

## 六、北京智能餐饮系统：AI 助手（DeepSeek 驱动）项目规划

### 6.1 引言
本章节规划一款由 DeepSeek 大模型驱动的智能餐饮助手系统，服务于北京地区的餐饮消费场景。系统深度集成店铺数据库、用户评价及实时地理位置，以自然语言交互替代传统多级筛选菜单，为用户提供个性化的“京城美食百事通”体验。

### 6.2 AI 助手定位
不同于通用 AI 聊天机器人，本系统中的 AI 助手具备以下核心特质：

- 深度绑定数据：基于底层店铺信息、实时评价和地理位置等结构化数据生成回答。
- 自然语言交互：理解复杂查询，直接返回精准结果，减少手动筛选步骤。
- 个性化服务：结合用户画像与偏好，实现千人千面的推荐体验。

### 6.3 核心功能模块设计

#### 6.3.1 交互入口：智能唤醒与引导
- 多态搜索词推导：进入界面时，AI 根据时间、天气、用户画像生成动态建议。
  - 早晨 8:00：想找找家附近的北京老字号早点铺吗？
  - 下雨天：今天适合足不出户，看看评分最高的火锅外送吧。
- 情景快捷卡片：提供“商务宴请”“深夜食堂”“遛娃首选”等一键入口，降低用户操作成本。

#### 6.3.2 核心引擎：深度语义理解与检索
- 精准问答（RAG）：基于店铺库与实时评价检索，不捏造事实。
  - 示例：帮我找海淀区人均 100 以内、有包间、不排队的川菜。
  - 解析：提取地理（海淀）、价格（<100）、设施（包间）、实时状态（低峰期）等约束，返回匹配结果。
- 口碑“显微镜”：过滤水军评价，提炼真实痛点，生成“避雷指南”。

#### 6.3.3 特色场景：智能路径规划
- CityWalk 美食线：结合北京旅游属性，输出“景点 + 餐饮”一站式路线方案。
- 多人聚餐中立人：输入多人位置与口味偏好后，计算距离中心点并推荐重合口味高的餐厅。

#### 6.3.4 菜单专家：点菜助手
- 避雷式点餐：基于评价数据自动生成“必点 Top3”与“可不点/性价比低”清单。
- 定制化组合：例如“3 人聚餐，想吃烤鸭并增加凉菜”，自动给出不浪费的菜品搭配方案。

### 6.4 用户偏好系统（数据库字段建议）
为了实现个性化推荐，建议在 Django 中通过扩展 `User` 模型或创建 `UserProfile` 实现偏好管理。

| 维度 | 字段 | 示例 |
|---|---|---|
| 基础偏好 | `flavor_pref` | 甜、咸、重辣、麻、清淡 |
| 饮食禁忌 | `dietary_restrictions` | 忌香菜、清真、海鲜过敏、素食 |
| 消费水平 | `avg_budget` | <50, 50-150, 150-500, 500+ |
| 常驻区域 | `frequent_areas` | 中关村、国贸、望京 |
| 场景权重 | `scene_weight` | 环境>味道 或 味道>环境 |
| 特殊标记 | `has_car` / `has_child` | 是否需要停车位、是否有儿童座椅需求 |

### 6.5 亮点功能（增强竞争力）
- “老北京”文化博主模式：推荐时附带北京饮食文化解释，提升内容价值与品牌调性。
- 实时“空位”预测：结合历史流量数据，预测排队趋势，给出“立即出发/在线取号”建议。
- 情绪点餐机：基于用户心情进行风格化推荐（甜品、辣味碳水、暖胃汤品等）。
- 智能发票/预订客服：连接商家信息，自动回复是否支持电子发票、是否可预订等问题。

### 6.6 技术实现方案（DeepSeek 集成）

#### 6.6.1 系统架构概述
系统建议采用三层架构：
- 数据层：MySQL 存储店铺、用户、评价；向量数据库（如 Milvus）存储评论向量与标签，支持语义检索。
- 处理层：DeepSeek 模型通过 Function Calling 调用系统查询接口，实现约束查询与解释生成。
- 展示层：前端采用“对话气泡 + 结构化卡片”融合展示，推荐语下方直接呈现可点击店铺卡片。

#### 6.6.2 Django 最简调用示例
DeepSeek API 兼容 OpenAI 协议，可直接通过 `openai` SDK 接入。

`settings.py`：
```python
DEEPSEEK_API_KEY = "YOUR_API_KEY"
DEEPSEEK_BASE_URL = "https://api.deepseek.com"
```

`services.py`：
```python
from openai import OpenAI
from django.conf import settings

def call_deepseek_ai(user_query, user_context=None):
    client = OpenAI(
        api_key=settings.DEEPSEEK_API_KEY,
        base_url=settings.DEEPSEEK_BASE_URL
    )

    context_str = f"用户信息：{user_context}" if user_context else ""
    response = client.chat.completions.create(
        model="deepseek-chat",  # 或 deepseek-reasoner
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT},
            {"role": "user", "content": f"{context_str}\n用户问题：{user_query}"}
        ],
        stream=False
    )
    return response.choices[0].message.content
```

#### 6.6.3 系统提示词（System Prompt）建议
为确保输出稳定性，建议将提示词固化为统一角色设定“京城味蕾管家”，并明确以下约束：
- 基于数据回答，不编造地址/电话。
- 优先遵循用户忌口、预算、距离等硬约束。
- 用户需求模糊时先反问关键条件（预算、区域、菜系）。
- 拒绝处理与餐饮、北京路线、食品健康无关的问题。

#### 6.6.4 `UserProfile` 示例模型
```python
from django.db import models
from django.contrib.auth.models import User

class UserProfile(models.Model):
    user = models.OneToOneField(User, on_delete=models.CASCADE)
    flavor_pref = models.CharField(max_length=200, blank=True, help_text="口味偏好，如甜、辣")
    dietary_restrictions = models.TextField(blank=True, help_text="饮食禁忌")
    avg_budget = models.CharField(max_length=20, blank=True, choices=[...])
    frequent_areas = models.CharField(max_length=200, blank=True)
    scene_weight = models.CharField(max_length=50, blank=True)
    has_car = models.BooleanField(default=False)
    has_child = models.BooleanField(default=False)
```

### 6.7 增强体验的最简落地逻辑
- 冷启动推荐词：按时间段返回 3 条建议问题（午餐、晚餐、夜宵）。
- 路线可视化：AI 产出文字路线，前端调用高德/百度地图 API 打点展示。
- 评论摘要缓存：离线批处理 `comments`，生成 `summary` 字段，降低在线推理成本并提升响应速度。

### 6.8 分阶段实施建议（Roadmap）
- Phase 1（2 周）：完成基础对话能力、条件解析、店铺检索与结果卡片展示。
- Phase 2（2~4 周）：接入用户偏好系统、口碑总结、菜单助手与场景卡片。
- Phase 3（4~6 周）：上线路径规划、多人聚餐中心点推荐、排队预测与运营看板。
- Phase 4（持续迭代）：优化模型效果、A/B 测试推荐策略、扩展“北京文化内容”能力。
