# 北京智能餐饮分析与推荐系统项目报告（完整版）

## 一、项目概述

### 1.1 项目名称
北京智能餐饮分析与推荐系统（DeepSeek 驱动）

### 1.2 项目背景
北京餐饮市场体量大、品类复杂、商圈分散，用户在“找店—比店—决策”过程中存在信息割裂、筛选成本高、踩雷风险高等问题。传统平台偏重信息罗列，缺乏针对场景与偏好的决策支持。本项目以“真实数据 + 智能推荐 + AI 助手 + 审核治理”为核心，构建可落地的餐饮决策平台。

### 1.3 项目目标
- 为用户提供快速、可信、可解释的就餐决策服务。
- 为商家提供规范化入驻和上架审核通道。
- 为管理员提供可追溯的数据治理和审核能力。
- 通过数据分析结果输出可执行建议，提升系统实用价值。

### 1.4 技术栈
- 后端：Django 5.2.10
- 数据库：MySQL 8.x
- 前端：Django Templates + HTML/CSS/JavaScript
- 图表：ECharts
- 地图：百度地图 API
- AI：DeepSeek（OpenAI 协议兼容）

## 二、系统实用性与业务价值

### 2.1 用户价值
- **决策效率提升**：通过筛选、推荐、AI问答减少重复检索。
- **踩雷风险降低**：综合评分、评论、点击、收藏等多维信息判断。
- **场景化体验**：商务宴请、遛娃、夜宵、平价午餐等场景快速决策。
- **可解释推荐**：每家店给出推荐理由，结果更可信。

### 2.2 商家价值
- **标准化入驻**：提交主体信息、地址、证照号、资质图片。
- **审核透明**：可查看审核状态、驳回原因并补充重提。
- **流程真实**：资质审核通过后才能申请店铺上架。

### 2.3 平台价值
- **双层审核机制**：商家资质审核 + 店铺上架审核。
- **数据质量提升**：降低虚假主体和低质店铺信息影响。
- **运营可持续**：形成用户、商家、管理员三端闭环。

## 三、系统亮点

### 3.1 AI 助手业务化
- DeepSeek + RAG 检索店铺上下文。
- 场景卡片内嵌偏好配置并直连对话。
- 输出结构化建议，强调“为什么推荐”。

### 3.2 推荐系统可解释与多样化
- 协同过滤 + 内容相似 + 价格匹配 + 热门补全。
- 按推荐理由分组展示，单理由控制推荐数量，避免同质化。

### 3.3 Top10 综合人气算法升级
- 引入评分、评论、点击、收藏四维加权。
- 弱化单一评论数量的误导，提高榜单实用性。

### 3.4 数据分析结果建议化
- 分析页不仅展示图表，还输出面向用户的行动建议。
- 强化“可读、可用、可执行”而非“只看图”。

### 3.5 商家入驻真实性
- 资质信息完整化（地址、证照、图片）。
- 审核结果可追溯，驳回原因可反馈，符合实际平台逻辑。

## 六、完整数据库设计

> 以下为系统核心表设计（基于当前项目模型），包含字段、含义、关系、约束与索引建议。

### 6.1 `food_shop_info`（店铺主表）

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | bigint PK | 自增主键 |
| `shop_id` | varchar(20) unique | 业务店铺ID |
| `shop_name` | varchar(100) | 店铺名称 |
| `longitude` / `latitude` | float | 经纬度 |
| `avg_price` | float | 人均消费 |
| `shop_score` | decimal(4,1) | 综合评分 |
| `comment_count` | int | 评论数 |
| `distance_desc` | varchar(50) null | 距离描述 |
| `michelin` | varchar(10) | 米其林标记 |
| `business_circle` | varchar(50) null | 商圈 |
| `food_type` | varchar(50) | 菜系 |
| `city` | varchar(20) | 城市 |
| `county` | varchar(20) | 区县 |
| `business_hours` | varchar(50) null | 营业时间 |
| `location` | varchar(100) | 详细地址 |
| `phone` | varchar(70) null | 电话 |

**约束/建议**
- 唯一键：`shop_id`
- 建议索引：`food_type`、`county`、`business_circle`、`shop_score`、`avg_price`

### 6.2 `food_comment_info`（评论表）

| 字段 | 类型 | 说明 |

| `sentiment` | varchar(10) null | 情感分类 |
| `sentiment_score` | float null | 情感分数 |

**建议索引**

|---|---|---|

| `user_id` | OneToOne -> auth_user.id | 商家账号 |

## 八、关键业务流程（闭环）

### 8.1 用户决策闭环
搜索/筛选 -> 推荐（理由）-> 详情（评论+地图）-> 收藏 -> 二次推荐更精准

### 8.2 AI 决策闭环
场景卡片设偏好 -> 保存偏好 -> 发起提问 -> AI结构化回答 -> 跳转店铺详情

### 8.3 商家入驻闭环
提交资质 -> 管理员审核资质 -> 通过后提交店铺上架 -> 管理员审核上架 -> 店铺入库可见

## 十、结论与展望

本系统从“数据展示型项目”升级为“决策服务型平台”，具备以下特征：
- 业务流程完整：用户、商家、管理员三端闭环。
- 数据可信可控：双层审核保障信息真实性。
- 结果可解释：推荐理由与建议输出增强可信度。
- 体验可落地：场景化AI与操作路径优化提升用户便利性。

后续可继续升级：
- 接入实时排队、天气、交通等特征；
- 引入A/B测试验证推荐策略；
- 扩展为多城市、多端（Web+小程序）部署。
